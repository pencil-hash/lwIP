# mem.c —— mem_malloc（）函数 ： 最核心的内存分配函数

## mem_malloc（）函数功能解析：

---

### 1. 分配数据包缓冲区（pbuf）

这是 `mem_malloc` 被调用**最频繁**的场景。

当网卡收到一个以太网帧时：
```
网卡硬件 → 触发中断 → 驱动程序需要内存存放这个包
                              ↓
                    调用 mem_malloc()
                              ↓
                    拿到一块内存 → 把数据包放进去
                              ↓
                    交给 IP 层处理
```

每一个在网络中传输的数据包，都需要一块内存来暂存。

---

### 2. 创建 TCP 连接（监听和连接）

当你的设备作为服务器，需要监听某个端口时：

```
应用程序调用 netconn_listen() 或 socket/bind/listen
         ↓
   需要创建 TCP 监听控制块
         ↓
   调用 mem_malloc() 分配 tcp_pcb_listen 结构体
         ↓
   控制块记录：我监听端口 80，谁来连我？
```

当有客户端连接进来时：

```
收到 SYN 包 → 需要为这个新连接创建 TCP 控制块
         ↓
   调用 mem_malloc() 分配 tcp_pcb 结构体
         ↓
   控制块记录：对方IP、对方端口、本方端口、发送/接收缓冲区、序列号...
```

**每个活跃的 TCP 连接，都对应一块通过 `mem_malloc` 分配的控制块内存。**

---

### 3. ARP 地址解析

当设备需要知道某个 IP 对应的 MAC 地址时：

```
需要发送数据给 192.168.1.100，但不知道它的 MAC 地址
         ↓
   创建 ARP 请求表项
         ↓
   调用 mem_malloc() 分配 ARP 表项
         ↓
   表项记录：IP 地址、MAC 地址（待填充）、超时时间
```

---

### 4. IP 数据包分片和重组

如果一个大 IP 数据包被拆分成了多个小片：

```
收到一个分片包 → 不是完整的 IP 包，需要等所有分片到齐
         ↓
   需要创建一个“分片重组缓冲区”
         ↓
   调用 mem_malloc() 分配重组结构体
         ↓
   把所有分片收集到这个缓冲区里，等拼完整了再交给上层
```

---

### 一张图总结

```
+------------------------------------------------------------------+
|                        lwIP 协议栈                                |
|                                                                  |
|  +---------+  +-------+  +-------+  +------+  +--------+        |
|  | TCP 连接 |  | UDP   |  | ARP   |  | IP   |  | 应用层  |        |
|  | 控制块  |  | 监听  |  | 缓存  |  | 分片  |  | 缓冲区  |        |
|  +----+----+  +---+---+  +--+----+  +--+---+  +---+----+        |
|       |           |          |          |          |              |
|       +-----------+----------+----------+----------+              |
|                           |                                      |
|                           v                                      |
|                  +-------------------+                           |
|                  |   mem_malloc()    |   ← 唯一的内存分配入口      |
|                  +-------------------+                           |
|                           |                                      |
|                           v                                      |
|                  +-------------------+                           |
|                  |     内存堆/池     |                            |
|                  +-------------------+                           |
+------------------------------------------------------------------+
```

---

### 一个具体例子：从收到数据到应用读取

跟踪一次完整的 TCP 数据接收，能清楚看到 `mem_malloc` 和 `mem_free` 如何配合：

```
1. 网卡中断: "有数据包来了！"
        ↓
2. 网卡驱动调用 mem_malloc()
   申请一块内存，把原始数据包存进去
        ↓
3. IP 层处理: "这是一个 TCP 包"
        ↓
4. TCP 层处理: "这是连接 #3 的数据"
   把数据拷贝到 TCP 连接的接收缓冲区
        ↓
5. 释放第 2 步分配的原始包内存
   mem_free()
        ↓
6. 应用程序调用 recv() 读取数据
   从 TCP 接收缓冲区拷贝数据给应用程序
        ↓
7. 缓冲区数据被读走后，空间释放
   mem_free()
```

**整个过程中**：
- `mem_malloc` 至少被调用了 **2 次**（分配包缓冲、分配 TCP 控制块/缓冲）
- `mem_free` 也被调用了 **2 次**（释放包缓冲、释放应用读走的数据空间）

---

所以，`mem_malloc()` 确实是 lwIP 中**分配内存的主要函数**，它是整个协议栈动态内存需求的统一入口。理解了它就等于理解了 lwIP 的“血液循环系统”。


## mem_malloc（）函数代码解析

---

### 函数签名与局部变量

```c
void * mem_malloc(mem_size_t size_in)
{
    mem_size_t ptr, ptr2, size;
    struct mem *mem, *mem2;
    ...
```

- `size_in`：协议栈上层请求的“我想用多少字节”。
- `ptr`、`ptr2`：块在堆中的**偏移量**（单位字节，从 `ram` 起始算起）。
- `size`：对齐并扩大后的实际需要大小。
- `mem`、`mem2`：指向控制块结构体的指针。

**关键概念**：  
每个内存块（无论已用还是空闲）前面都有一个 `struct mem` 控制头，里面存着 `next`、`prev`、`used`。用户拿到的指针是**跳过控制头**之后的数据区地址。

---

### 第一步：拒绝 0 字节请求

```c
if (size_in == 0) {
    return NULL;
}
```

**为什么？**  
分配 0 字节没有意义，而且可能导致后续计算出现奇怪边界情况，直接返回 NULL 表示“失败”。

---

### 第二步：调整请求大小（对齐 + 最小块限制）

```c
size = (mem_size_t)LWIP_MEM_ALIGN_SIZE(size_in);
if (size < MIN_SIZE_ALIGNED) {
    size = MIN_SIZE_ALIGNED;
}
#if MEM_OVERFLOW_CHECK
size += MEM_SANITY_REGION_BEFORE_ALIGNED + MEM_SANITY_REGION_AFTER_ALIGNED;
#endif
if ((size > MEM_SIZE_ALIGNED) || (size < size_in)) {
    return NULL;
}
```

### 2.1 对齐处理

`LWIP_MEM_ALIGN_SIZE(size_in)` 把请求大小**向上取整**到 `MEM_ALIGNMENT` 的倍数。  
假设 `MEM_ALIGNMENT=4`，你请求 7 字节，它会变成 8。  
**为什么？** 很多 CPU 访问 4 字节数据时，地址必须被 4 整除，否则硬件异常。对齐能保证后续返回的数据指针满足要求。

### 2.2 最小块限制

`MIN_SIZE_ALIGNED` 是预定义的最小块大小（默认 12 字节，对齐后可能更大）。  
如果对齐后的大小比它还小，就**强行拉高到最小块**。  
**为什么？** 假如你只申请 1 字节，对齐后可能只有 4 字节。如果以后释放这个块，它会变成一块极小的空闲区，再也无法满足任何其它请求（因为每个块至少需要放一个 `struct mem` 头部），成为**永久碎片**。限制最小尺寸就是为了防止这种浪费。

### 2.3 溢出检查额外开销

如果开启了 `MEM_OVERFLOW_CHECK`，每个块前后会多出几个字节的“哨兵”空间，用来检测写越界。这里 `size` 要加上这些哨兵区的大小，保证实际分配的空间能容纳哨兵。

### 2.4 大小合法性检查

- `size > MEM_SIZE_ALIGNED`：对齐后的需求比整个堆还大，肯定分配不了，返回 NULL。
- `size < size_in`：由于计算过程中可能发生整数溢出（比如 `size_in` 是 `0xFFFF`，加上溢出检测字节后溢出了），如果出现这种 `size` 反而变小的情况，也返回 NULL。

---

### 第三步：加锁并开始搜索

```c
sys_mutex_lock(&mem_mutex);
LWIP_MEM_ALLOC_PROTECT();
```

这两行保护堆的并发访问。在有操作系统的环境，用互斥锁；裸机环境下可能是关中断。总之，保证分配过程不会被其他任务打断。

---

### 第四步：首次适配（First Fit）搜索循环

```c
for (ptr = mem_to_ptr(lfree);
     ptr < MEM_SIZE_ALIGNED - size;
     ptr = ptr_to_mem(ptr)->next)
{
    mem = ptr_to_mem(ptr);
    ...
```

### 4.1 循环初始化

- `lfree` 是一个全局指针，指向**地址最低的空闲块**（可能是 `ram_end` 如果完全没有空闲块）。
- `mem_to_ptr(lfree)` 将这个指针转换成它在堆中的偏移量 `ptr`。

### 4.2 循环条件

`ptr < MEM_SIZE_ALIGNED - size`  
**意思**：当前块的偏移量必须 ≤（堆总大小 - 请求大小）。  
如果 `ptr` 太大，即使这一块后面的空间全给你，也凑不够 `size` 字节，后面就更不可能有了，所以停止搜索。

### 4.3 循环步进

`ptr = ptr_to_mem(ptr)->next`  
每次检查完当前块后，跳到它链表中的**下一块**继续检查。  
这就是沿着双向链表向后走。

---

### 第五步：检查当前空闲块是否够大

```c
if ((!mem->used) &&
    (mem->next - (ptr + SIZEOF_STRUCT_MEM)) >= size)
```

这个 `if` 判断必须两个条件都满足：

1. `!mem->used` —— 这一块是空闲的。
2. 可用数据区 ≥ `size`

### 可用数据区怎么算？

一块内存的总大小 = 从它的控制头到下一块控制头之间的字节数。  
这个值等于 `mem->next - ptr`（`ptr` 是本块控制头偏移）。  
但是控制头本身要占 `SIZEOF_STRUCT_MEM` 字节，所以**用户可用的数据区大小** = `mem->next - ptr - SIZEOF_STRUCT_MEM`。

**因此条件 2 写成**：  
`mem->next - (ptr + SIZEOF_STRUCT_MEM) >= size`  
（把括号展开就一样了）

---

### 第六步：找到后，要切分吗？

这里出现一个大的 `if-else`：

### 情况 A：剩余空间足够再分割（创建 mem2）

```c
if (mem->next - (ptr + SIZEOF_STRUCT_MEM) >=
    (size + SIZEOF_STRUCT_MEM + MIN_SIZE_ALIGNED))
```

**逻辑**：如果分配 `size` 后剩下的空间，还装得下另一个控制头（`SIZEOF_STRUCT_MEM`）+ 最小数据区（`MIN_SIZE_ALIGNED`），我们就**切分**，避免浪费。

**切分步骤**：

1. 计算新空闲块控制头的位置 `ptr2`：  
   `ptr2 = ptr + SIZEOF_STRUCT_MEM + size`  
   （紧跟在分配出去的数据区后面）

2. 在 `ptr2` 处创建新控制块 `mem2`：  
   ```c
   mem2 = ptr_to_mem(ptr2);
   mem2->used = 0;             // 新块是空闲的
   mem2->next = mem->next;     // 继承原块的下一块指针
   mem2->prev = ptr;           // 新块的前一块是当前 mem
   ```

3. 修改当前块 `mem` 的信息：  
   ```c
   mem->next = ptr2;           // 当前块的下一个变成 mem2
   mem->used = 1;              // 当前块标记为已用
   ```

4. 如果 mem2 后面还有块（即 `mem2->next != MEM_SIZE_ALIGNED`），需要更新那个块的 `prev` 指针为 `ptr2`。

**切分后的布局**：  
```
[mem: used=1] -> [数据区 size] -> [mem2: used=0, next=原next, prev=ptr] -> ...
```

### 情况 B：剩余空间不够再分割

```c
else {
    mem->used = 1;
    // 记录统计使用的空间 = 整块大小（到下一块之前）
    MEM_STATS_INC_USED(used, mem->next - mem_to_ptr(mem));
}
```

**逻辑**：剩余空间太小，不能单独成为一个有效块（放不下控制头+最小数据）。与其留下一个永久无用的碎片，不如把整块都分配给这次请求。用户虽然只用了 `size`，但额外多给了一点点空间。

**标记**：直接把 `mem->used = 1`，不创建 `mem2`。

---

### 第七步：更新 `lfree`（最低空闲块指针）

```c
if (mem == lfree) {
    struct mem *cur = lfree;
    while (cur->used && cur != ram_end) {
        cur = ptr_to_mem(cur->next);
    }
    lfree = cur;
    LWIP_ASSERT(...);
}
```

**什么情况下触发？**  
当刚刚分配出去的块**恰好就是 `lfree`** 时，`lfree` 现在指向了一个已用块，这破坏了“`lfree` 始终指向最低空闲块”的约定，必须更新。

**更新方法**：  
从原来 `lfree` 的位置开始，沿着链表向后找，直到遇到第一个 `used == 0` 的块（或者遇到堆末尾 `ram_end`），让 `lfree` 指向它。  
因为 `lfree` 总是从低地址开始找，所以找到的一定是新的“最低空闲块”。

---

### 第八步：解锁，返回用户指针

```c
LWIP_MEM_ALLOC_UNPROTECT();
sys_mutex_unlock(&mem_mutex);
...
return (u8_t *)mem + SIZEOF_STRUCT_MEM + MEM_SANITY_OFFSET;
```

### 为什么返回的是这个地址？

- `(u8_t *)mem` 是本块控制头的起始地址。  
- `+ SIZEOF_STRUCT_MEM` 跳过控制头本身，到达**数据区的开头**。  
- 如果开启了溢出检测，再 `+ MEM_SANITY_OFFSET` 跳过前面的哨兵区域。

**一句话**：用户只能动数据区，绝对不能碰控制头，不然链表就被破坏了。

---

### 第九步：如果遍历完都没找到合适块

```c
MEM_STATS_INC(err);
...
return NULL;
```

堆里没有足够大的连续空闲空间，分配失败，返回 NULL。协议栈上层必须处理这个 NULL，通常是丢弃数据包或拒绝连接。

---

### 五个核心问题直接回答

### 1. 怎么对齐大小、设最小块限制？

- **对齐**：`LWIP_MEM_ALIGN_SIZE(size_in)` 将请求大小向上取整到 `MEM_ALIGNMENT`（通常是 4 或 8）的倍数。  
- **最小块限制**：如果对齐后小于 `MIN_SIZE_ALIGNED`（例如 12），就强制设为 `MIN_SIZE_ALIGNED`。避免未来释放后产生无法使用的微小空闲块（碎片）。

### 2. 首次适配（first fit）的循环怎么走？

- 从 `lfree`（最低空闲块）开始，取得它的偏移量 `ptr`。
- 循环条件：`ptr < MEM_SIZE_ALIGNED - size`，即当前偏移量还没大到剩余空间不可能满足请求。
- 步进：`ptr = ptr_to_mem(ptr)->next`，即沿着链表走到下一块。
- 在循环体内，检查当前块是否 `!used` 且可用数据区 ≥ `size`，一旦找到就处理，找不到就继续。

### 3. 什么时候会切分一个大块（创建 `mem2`）？

当找到的空闲块的可用数据区 **大于等于** `size + SIZEOF_STRUCT_MEM + MIN_SIZE_ALIGNED` 时。

也就是说，分配 `size` 后剩下的空间，还能装得下至少一个控制头 + 最小数据量。这时就把这个大块切成两部分：
- 第一部分（`mem`）分配出去，`used=1`。
- 第二部分（`mem2`）成为新的空闲块，`used=0`，插入链表。

如果剩余空间不够这个条件，就不切分，直接把整块分配出去。

### 4. 什么时候更新 `lfree`？

当本次分配出去的块 `mem` **正好等于 `lfree`** 时。  
因为 `lfree` 不能再指向已用块，所以从它开始向后扫描，找到第一个空闲块作为新的 `lfree`。

这保证了下次分配时，`lfree` 仍然指向地址最低的空闲块，加速搜索。

### 5. 返回给用户的是 `(u8_t*)mem + SIZEOF_STRUCT_MEM`，为什么？

因为 `mem` 指向的是控制块 `struct mem` 的开头，里面是 `next`、`prev`、`used` 等管理信息，不是给用户存数据的。  
向后跳过 `SIZEOF_STRUCT_MEM` 个字节，才到达紧跟在控制块后面的**用户数据区**。  
这个指针就是用户可以自由读写的空间。

（如果有溢出检查，还会多跳过一个哨兵区偏移量 `MEM_SANITY_OFFSET`，逻辑相同。）

---

掌握这五点，再配合上面的逐段讲解，你就能彻底搞懂 lwIP 核心内存分配的原理了。建议拿一张纸，画出一个初始堆，然后模拟一次 `malloc(100)` 和 `malloc(200)`，手动标出 `ptr`、`ptr2`、`next`、`prev` 的变化，会非常直观。


##  mem_malloc（）函数图形讲解
---

### 场景设定

假设我们的堆总共 **256 字节**，`MEM_ALIGNMENT = 4`，`SIZEOF_STRUCT_MEM = 8`（控制头大小），`MIN_SIZE_ALIGNED = 12`。

### 初始状态（mem_init 刚执行完）

`lfree` 指向第一个空闲块，`ram_end` 是末尾哨兵。

```
地址: 0                                                            256
     +--------+---------------------------------------------------+------+
     | 空闲块  |                完全空闲空间                          | 哨兵 |
     | next=256|                                                   |used=1|
     | prev=0  |                                                   |      |
     | used=0  |                                                   |      |
     +--------+---------------------------------------------------+------+
     ^                                                             ^
     lfree                                                         ram_end
     ptr=0
```

---

### 现在调用 `mem_malloc(60)` 请求 60 字节

### 第 1 步：调整大小

- `size_in = 60`
- `LWIP_MEM_ALIGN_SIZE(60)` = 60 向上对齐到 4 的倍数 = **60**（刚好对齐）
- 60 ≥ `MIN_SIZE_ALIGNED(12)`，不需要调整
- 最终 `size = 60`

---

### 第 2 步：进入搜索循环

- `ptr = mem_to_ptr(lfree) = 0`（从偏移 0 开始）
- 检查条件：`ptr(0) < 256 - 60` ？是，继续
- `mem = ptr_to_mem(0)`（指向地址 0 的控制头）

```
偏移: 0                                                            256
     +--------+---------------------------------------------------+------+
     | 空闲块  |                完全空闲空间                          | 哨兵 |
     | next=256|                                                   |used=1|
     | prev=0  |                                                   |      |
     | used=0  |                                                   |      |
     +--------+---------------------------------------------------+------+
     ^
     ptr=0, mem
```

---

### 第 3 步：检查这个块

- `!mem->used` → `!0` → **TRUE**（空闲）
- 可用数据区 = `mem->next(256) - (ptr(0) + 8)` = **248 字节**
- 248 ≥ 60？**YES！**

---

### 第 4 步：要切分吗？

- 条件：`可用空间(248) >= size(60) + SIZEOF_STRUCT_MEM(8) + MIN_SIZE_ALIGNED(12)`
- 248 ≥ 80？**YES！** 切分！

#### 切分操作：

新空闲块的偏移：`ptr2 = ptr(0) + 8 + 60 = 68`

```
偏移: 0        8              68                                       256
     +--------+---------------+--------+-----------------------------+------+
     | mem    | 用户数据(60B)  | mem2   |         剩余空闲              | 哨兵 |
     | next=68|               | next=? |                              |used=1|
     | prev=0 |               | prev=? |                              |      |
     | used=1 |               | used=? |                              |      |
     +--------+---------------+--------+-----------------------------+------+
     ^                        ^
     mem                      mem2
```

#### 设置 mem2 的指针：

- `mem2->next = mem->next` → 原来 mem 的 next 是 256，所以 `mem2->next = 256`
- `mem2->prev = ptr` → `mem2->prev = 0`
- `mem2->used = 0`

#### 修改 mem 的指针：

- `mem->next = ptr2` → `mem->next = 68`
- `mem->used = 1`

#### 更新原下一块(哨兵)的 prev：

- `mem2->next = 256`，不是 `MEM_SIZE_ALIGNED`？
- 等一下，`256 == MEM_SIZE_ALIGNED`，所以**哨兵不需要更新 prev**。

---

### 第 5 步：切分完成后

```
偏移: 0        8              68                                       256
     +--------+---------------+--------+-----------------------------+------+
     | mem    | 用户数据(60B)  | mem2   |         剩余空闲              | 哨兵 |
     | next=68|               | next=256|                             |used=1|
     | prev=0 |               | prev=0 |                              |      |
     | used=1 |               | used=0 |                              |      |
     +--------+---------------+--------+-----------------------------+------+
     ^                        ^
     ptr=0                    ptr2=68
```

---

### 第 6 步：更新 lfree

- 检查 `mem == lfree`？
- `mem` 在偏移 0，`lfree` 也是偏移 0 → **YES**
- 需要更新 `lfree`

**更新过程**：

- `cur->used = 1`（已用）且 `cur != ram_end` → 继续找


# mem.c —— mem_free（）负责释放内存的核心函数
好的，我们先从“这个函数在协议栈里是干什么的”这个角度建立整体认知，然后再逐段拆解代码。

---

## 一、从协议功能方面看 `mem_free`

### 1. 它在整个 lwIP 中的角色

lwIP 是一个网络协议栈，它需要频繁地做两件事：

- **分配内存**：收到数据包、创建连接、发送数据时
- **释放内存**：数据包处理完、连接关闭、发送完成时

`mem_free` 就是负责“释放”的那个核心函数。

**类比**：如果把 lwIP 比作一个快递中转站，`mem_malloc` 是从仓库里领一个空箱子来装快递，`mem_free` 就是把用完的空箱子还回仓库，并且**把相邻的空箱子自动粘在一起，形成一个更大的空箱子**，方便下次装更大的快递。

### 2. 它在什么场景下被调用

| 协议场景 | 谁调用 `mem_free` |
|---------|------------------|
| TCP 连接关闭 | 释放 TCP 控制块（tcp_pcb） |
| UDP 数据包处理完 | 释放 pbuf（数据包缓冲区） |
| ARP 表项过期 | 释放 ARP 表项内存 |
| IP 分片重组超时 | 释放分片缓冲区 |
| 内存池回收 | 把池中的元素归还 |

**一句话**：只要协议栈里某块动态内存不再需要了，最终都会调用 `mem_free` 来归还。

### 3. 它要解决的三个核心问题

| 问题 | `mem_free` 的做法 |
|------|------------------|
| **如何知道这块内存属于谁？** | 从用户指针反向推导出控制头（`struct mem`），里面记录了这块内存的大小和链表关系 |
| **如何防止内存碎片？** | 释放后立即调用 `plug_holes`，检查相邻块是否空闲，如果是就合并成一个大块 |
| **如何防止 bug 破坏整个堆？** | 在释放前做多层“安检”：空指针、对齐、越界、双重释放、链表完整性 |

---

## 二、从代码学习方面逐段详解

我们先把代码分成 8 个段落，一段一段看。忽略调试宏和统计宏（`LWIP_DEBUGF`、`MEM_STATS_*`），只关注核心逻辑。

---

### 段落 0：辅助函数 `mem_link_valid`

在 `mem_free` 之前，我们先看这个辅助函数，因为它被 `mem_free` 用来做链表完整性检查。

```c
static int
mem_link_valid(struct mem *mem)
{
  struct mem *nmem, *pmem;
  mem_size_t rmem_idx;
  rmem_idx = mem_to_ptr(mem);       // 本块在堆中的偏移量
  nmem = ptr_to_mem(mem->next);     // 下一块的指针
  pmem = ptr_to_mem(mem->prev);     // 上一块的指针
  if ((mem->next > MEM_SIZE_ALIGNED) || (mem->prev > MEM_SIZE_ALIGNED) ||
      ((mem->prev != rmem_idx) && (pmem->next != rmem_idx)) ||
      ((nmem != ram_end) && (nmem->prev != rmem_idx))) {
    return 0;  // 链表损坏
  }
  return 1;     // 链表正常
}
```

**它在检查什么？**

```
假设链表是这样的：
    +-------+    +-------+    +-------+
    | pmem  | -> | mem   | -> | nmem  |
    | prev A|    | prev B|    | prev C|
    | next B|    | next C|    | next D|
    +-------+    +-------+    +-------+
```

这个函数检查三点：

1. **`mem->next` 和 `mem->prev` 不能超过堆的总大小**  
   如果 `next` 指向 300，但堆只有 256，说明数据被写坏了。

2. **前向链接检查**：  
   如果 `mem->prev` 不等于自己（即不是堆的第一个块），那么**前一块的 `next` 必须指向本块**。  
   `pmem->next != rmem_idx` 就是“前一站说下一站是我，但实际不是我”。

3. **后向链接检查**：  
   如果 `mem->next` 不是哨兵节点（`ram_end`），那么**下一块的 `prev` 必须指向本块**。  
   `nmem->prev != rmem_idx` 就是“下一站说前一站是我，但实际不是我”。

如果任何一个条件不满足，返回 0（无效），否则返回 1（有效）。

### 段落 1：函数签名与变量声明

```c
void
mem_free(void *rmem)
{
  struct mem *mem;
  LWIP_MEM_FREE_DECL_PROTECT();
```

- `rmem`：用户想释放的指针（之前 `mem_malloc` 返回的那个）。
- `mem`：我们将从 `rmem` 反推出的控制块指针。
- `LWIP_MEM_FREE_DECL_PROTECT()`：准备好并发保护的“保险丝”（可能是声明一个保存中断状态的变量，也可能什么都不做，取决于配置）。

---

### 段落 2：第一道安检——空指针

```c
  if (rmem == NULL) {
    LWIP_DEBUGF(MEM_DEBUG | LWIP_DBG_TRACE | LWIP_DBG_LEVEL_SERIOUS, ("mem_free(p == NULL) was called.\n"));
    return;
  }
```

- C 语言中 `free(NULL)` 是合法的（什么都不做）。
- lwIP 也采用同样策略：传 `NULL` 进来，直接返回，不报错不崩溃。

---

### 段落 3：第二道安检——对齐

```c
  if ((((mem_ptr_t)rmem) & (MEM_ALIGNMENT - 1)) != 0) {
    LWIP_MEM_ILLEGAL_FREE("mem_free: sanity check alignment");
    MEM_STATS_INC_LOCKED(illegal);
    return;
  }
```

**检查逻辑**：

- `MEM_ALIGNMENT` 通常是 4。
- `MEM_ALIGNMENT - 1` = 3，二进制是 `0b11`。
- `任何地址 & 3`：取地址的最低 2 位。
- 如果最低 2 位不是 0，说明地址不是 4 的倍数 → **没对齐**。

**为什么这很重要？**

我们分配时保证所有返回的指针都是对齐的。如果用户传入的指针没对齐，一定不是我们分配出去的，可能是传错了、野指针、或者内存被破坏了。

**图示**：

```
地址: 0x1000 (对齐 ✓)     0x1003 (没对齐 ✗)
      ────                  ────
      可以被4整除            不能被4整除
```

---

### 段落 4：从用户指针反推控制头

```c
  mem = (struct mem *)(void *)((u8_t *)rmem - (SIZEOF_STRUCT_MEM + MEM_SANITY_OFFSET));
```

**这是释放操作中最关键的一行代码。**

**分配时的公式**（`mem_malloc` 返回语句）：
```
返回给用户的指针 = (u8_t*)控制头 + SIZEOF_STRUCT_MEM + MEM_SANITY_OFFSET
```

**释放时的逆公式**（这一行）：
```
控制头指针 = (u8_t*)用户指针 - SIZEOF_STRUCT_MEM - MEM_SANITY_OFFSET
```

**用我们之前的数字画图**（SIZEOF_STRUCT_MEM=8, 无溢出检查）：

```
偏移:    68               76
         +---------------+------------------+
         | 控制头(8字节)  | 用户数据区        |
         | next           |                  |
         | prev           |                  |
         | used           |                  |
         +---------------+------------------+
         ^               ^
         |               |
       mem 指针        rmem (用户拿到的指针)
       (偏移68)        (偏移76)

mem = 76 - 8 = 68  ← 反推成功
```

**类型转换解读**：

- `(u8_t *)rmem`：把用户指针变成字节指针，这样才能做字节级别的减法。
- `- (SIZEOF_STRUCT_MEM + MEM_SANITY_OFFSET)`：往前退回控制头的开头。
- `(void *)`：先转成 `void*`，消除编译器的对齐警告。
- `(struct mem *)`：最终转成控制块指针。

---

### 段落 5：第三道安检——范围检查

```c
  if ((u8_t *)mem < ram || (u8_t *)rmem + MIN_SIZE_ALIGNED > (u8_t *)ram_end) {
    LWIP_MEM_ILLEGAL_FREE("mem_free: illegal memory");
    MEM_STATS_INC_LOCKED(illegal);
    return;
  }
```

**两个边界条件**：

| 条件 | 含义 |
|------|------|
| `mem < ram` | 控制头地址比堆起始还小 → 不在堆内 |
| `rmem + MIN_SIZE_ALIGNED > ram_end` | 用户数据区加最小块大小超出了堆末尾 → 超出堆范围 |

**图示**：

```
ram (堆起始)                                    ram_end (堆末尾)
  ↓                                               ↓
  +--------+--------+--------+-----+--------+-----+
  | 块1    | 块2    | 块3    | ... | 块n    |哨兵 |
  +--------+--------+--------+-----+--------+-----+
  ↑                                            ↑
  合法区域                                     合法边界
        ↑                          ↑
    mem必须在这之间           rmem+MIN_SIZE_ALIGNED
                              不能超过这里
```

---

### 段落 6：溢出检查（可选）

```c
#if MEM_OVERFLOW_CHECK
  mem_overflow_check_element(mem);
#endif
```

如果开启了溢出检查，会去读控制块后面哨兵区域的值，看是不是还是 `0xcd`。如果被改了，说明用户写了越界数据，触发断言报警。

---

### 段落 7：加锁

```c
  LWIP_MEM_FREE_PROTECT();
```

从现在开始，堆的链表不能被中断或其他任务修改。  
这就像进仓库先锁门，防止别人同时搬箱子导致混乱。

---

### 段落 8：第四道安检——双重释放

```c
  if (!mem->used) {
    LWIP_MEM_ILLEGAL_FREE("mem_free: illegal memory: double free");
    LWIP_MEM_FREE_UNPROTECT();
    MEM_STATS_INC_LOCKED(illegal);
    return;
  }
```

- `mem->used` 只有两个可能值：0（空闲）或 1（已用）。
- 如果已经是 0，说明这块内存已经被释放过了。
- **双重释放是 C 语言中最危险的 bug 之一**，因为它会导致链表混乱，后续分配可能把同一块内存给两个人用。

**这里先解锁再返回**，因为既然要拒绝操作，就得把仓库门重新打开。

---

### 段落 9：第五道安检——链表完整性

```c
  if (!mem_link_valid(mem)) {
    LWIP_MEM_ILLEGAL_FREE("mem_free: illegal memory: non-linked: double free");
    LWIP_MEM_FREE_UNPROTECT();
    MEM_STATS_INC_LOCKED(illegal);
    return;
  }
```

调用我们前面讲过的 `mem_link_valid` 函数，检查这个块的 `next`/`prev` 关系是否完整。

**什么时候会触发这个？**

- 用户写了越界数据，破坏了控制头的 `next` 或 `prev`。
- 双重释放的另一种表现（链表已经被上次释放破坏了）。
- 野指针释放。

---

### 段落 10：正式释放——标记为空闲

```c
  mem->used = 0;
```

只需要改这一个标志位。从这一瞬间开始，这块内存就变成了“空闲块”，可以被后续的 `mem_malloc` 重新分配。

---

### 段落 11：更新最低空闲指针

```c
  if (mem < lfree) {
    lfree = mem;
  }
```

- `lfree` 始终指向**地址最低的空闲块**。
- 如果刚释放的这个块地址比当前 `lfree` 还低，就更新 `lfree`。
- 这样下次分配时，能从更低的地址开始搜索，提高效率。

**图示**：

```
释放前：
    lfree → 块2(空闲, 偏移100)
    块1(已用, 偏移0)     块3(已用, 偏移200)

释放块1后：
    lfree → 块1(空闲, 偏移0)    ← 更新了！
    块2(空闲, 偏移100)         块3(已用, 偏移200)
```

---

### 段落 12：更新统计

```c
  MEM_STATS_DEC_USED(used, mem->next - (mem_size_t)(((u8_t *)mem - ram)));
```

这段只在启用统计时才有效果，计算本块的总大小并从“已用内存”统计中减去。

---

### 段落 13：核心——合并相邻空闲块

```c
  plug_holes(mem);
```

这是 `mem_free` 的**灵魂**。它会：

1. **向前合并**：如果本块的下一块也是空闲，就把它们合并。
2. **向后合并**：如果本块的上一块也是空闲，也合并。

**为什么这很重要？**

如果不合并，多次分配/释放后，堆会变成“瑞士奶酪”——很多小空洞，即使总空闲空间很大，却找不到一块连续的大空间来满足稍大的请求。

`plug_holes` 的具体逻辑我们下次可以单独展开，但你现在先记住它的作用：**把相邻的空闲块粘在一起，防止碎片化。**

---

### 段落 14：完整性检查与解锁

```c
  MEM_SANITY();
#if LWIP_ALLOW_MEM_FREE_FROM_OTHER_CONTEXT
  mem_free_count = 1;
#endif
  LWIP_MEM_FREE_UNPROTECT();
```

- `MEM_SANITY()`：可选的整体堆完整性校验，遍历整个链表检查是否有异常。
- `mem_free_count = 1`：如果配置了允许中断上下文释放，这里标记“有释放发生过”，通知可能在并发运行的 `mem_malloc` 重新检查。
- `LWIP_MEM_FREE_UNPROTECT()`：解锁，仓库门重新打开。

---

## 总结：`mem_free` 的完整流程图

```
用户调用 mem_free(指针)
       │
       ▼
  ┌─ 安检1: 是否NULL？ ──→ 是 → 直接返回
  │     否
  │     ▼
  ├─ 安检2: 是否对齐？ ──→ 否 → 记录非法统计，返回
  │     是
  │     ▼
  ├─ 反推控制头: mem = 指针 - 控制头大小
  │     ▼
  ├─ 安检3: 控制头是否在堆范围内？ ──→ 否 → 返回
  │     是
  │     ▼
  ├─ 安检4(可选): 溢出检查，哨兵值完整？
  │     ▼
  ├─ 加锁
  │     ▼
  ├─ 安检5: mem->used == 1？ ──→ 否(双重释放) → 解锁，返回
  │     是
  │     ▼
  ├─ 安检6: 链表是否完整？ ──→ 否 → 解锁，返回
  │     是
  │     ▼
  ├─ 标记为空闲: mem->used = 0
  │     ▼
  ├─ 更新 lfree（如果需要）
  │     ▼
  ├─ 合并相邻空闲块: plug_holes(mem)
  │     ▼
  ├─ 完整性检查: MEM_SANITY()
  │     ▼
  └─ 解锁，完成
```

这层层安检的设计，正是为了在嵌入式资源受限环境下，尽早发现内存越界、野指针、双重释放等常见 bug，防止它们蔓延导致整个协议栈崩溃。
